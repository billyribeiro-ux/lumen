# ADR-004: Authentication — Better Auth over Lucia, Auth.js, Clerk, and custom

**Status:** Accepted
**Date:** 2026-04-24
**Deciders:** @billyribeiro-ux
**Phase:** 0 — Foundation
**Tags:** `auth`, `security`, `foundation`

---

## Context

Authentication for Lumen must support:

- **Email + password** with modern hashing (Argon2id, not bcrypt).
- **Passkeys (WebAuthn)** as the preferred factor — passwordless is the future and the present for keyboard-first power users.
- **TOTP 2FA** for accounts that opt in.
- **OAuth** with Google and GitHub at minimum, with room for more providers later.
- **Magic links** for friction-free sign-in.
- **Server-side session storage** in Postgres — no JWT, no stateless tokens, no third-party identity service holding the session.
- **Direct ownership of user data.** Users belong to Lumen, not to a vendor's identity service.
- **Drizzle adapter** so user / session / account tables live in our schema and migrate alongside the rest of the data model.
- **Framework-agnostic core** so a future Tauri-only auth flow or a future internal CLI tool can use the same library.

Four candidates were evaluated: Better Auth, Lucia, Auth.js (NextAuth), and Clerk. "Roll our own" was considered as the do-nothing alternative.

---

## Decision

**We will use Better Auth 1.6.x as Lumen's authentication library**, adapted to Drizzle and Postgres, with the following plugins enabled: `passkey`, `two-factor` (TOTP), `magic-link`, and the OAuth providers `google` and `github`.

Concretely:

- The Better Auth config lives in `src/lib/server/auth.ts` and exports a singleton `auth` instance.
- Required tables (`user`, `session`, `account`, `verification`) are defined in `src/lib/server/db/schema/auth.ts` using Drizzle's `pgTable` so migrations are generated and reviewed like any other schema change.
- Session resolution happens in `hooks.server.ts` and writes to `event.locals.user` and `event.locals.session`. Every downstream handler reads from `locals` only — never from raw cookies.
- Sign-in routes are `+page.server.ts` form actions that delegate to Better Auth handlers; they degrade to no-JS form posts gracefully.
- The Better Auth client is **not** used on the browser by default. Server-rendered pages write the session cookie; client-side calls go through Lumen-defined remote functions, not Better Auth's client SDK. This keeps the auth surface server-controlled.

---

## Consequences

### Positive

- **One library, all factors.** Passkeys, TOTP, OAuth, magic links — all in a single, unified configuration. No stitching together `simplewebauthn` + a TOTP library + an OAuth library + a session manager.
- **Sessions live on our database.** Postgres is the source of truth for who is signed in. Sign-out, session revocation, "sign me out everywhere" — all are SQL writes.
- **Framework-agnostic.** Better Auth runs in any JavaScript environment. The same library powers SvelteKit handlers today, Tauri local commands tomorrow, and any future Node-only worker that needs to authenticate a user.
- **Drizzle adapter is first-class.** Schema and migrations are owned by us, generated by `drizzle-kit`, and migrate in lock-step with the rest of the application schema.
- **No vendor lock-in on identity data.** Users, credentials, and sessions live in Lumen's Postgres. Migrating off Better Auth, if it ever became necessary, is a library swap — not a data migration.
- **Active development and a clear governance model.** Releases are frequent, breaking changes are flagged, and the project's design document is unusually clear for an auth library.

### Negative

- **Younger than Auth.js.** Better Auth's 1.x line stabilized in late 2024–2025; some edge plugins (e.g. enterprise SSO via SAML) lag behind Auth.js. We do not need them at v1.0.0.
- **Documentation is improving but not yet exhaustive.** Some advanced configurations (custom session enrichment, complex consent flows) require reading source. Acceptable cost for a library this central.
- **Plugin surface is broad.** Each plugin we enable expands the test matrix. Passkey + TOTP + magic-link + 2 OAuth providers is already non-trivial. Test discipline (Phase 14) must cover every sign-in path.

### Neutral

- Better Auth's session strategy is server-side by default, which matches our requirement exactly. No tuning needed.
- The library does not include a UI. Lumen builds its own auth UI on top of Bits UI primitives (ADR-007). This is the right separation; auth UIs are too brand-specific to import.

## Alternatives Considered

### Alternative 1: Lucia

**Pros:**
- Excellent design philosophy: small, framework-agnostic, focused on session management.
- Tight integration with Drizzle and Postgres.
- Famous for clean, readable source.

**Cons:**
- **Project sunset announced 2024.** The maintainer announced Lucia would be archived in favor of educational guides on building your own auth. The library still works, but it is no longer maintained — the 10-year longevity test fails.
- Lucia v3 was session-only; passkeys, TOTP, OAuth, and magic links were all out-of-band, requiring separate libraries we'd integrate ourselves.

**Why rejected:** A sunsetted library is a liability we cannot accept for the core auth layer.

### Alternative 2: Auth.js (NextAuth)

**Pros:**
- Largest installed base in the JavaScript auth space.
- Many providers built in.
- SvelteKit adapter exists.

**Cons:**
- Designed primarily for Next.js. The SvelteKit adapter is a port that lags the core release cadence.
- Default session strategy is JWT — the opposite of what we want. Database sessions require additional configuration and the JWT fallbacks add cognitive overhead.
- Passkey support is not first-class; it's a community plugin.
- The Auth.js 5.x rewrite churned the API significantly through 2024; configuration patterns from year-old tutorials no longer apply.
- Tight conceptual coupling to Next's middleware model creates friction in non-Next environments.

**Why rejected:** The SvelteKit experience is second-class and the JWT-default + passkey-second-class profile is the inverse of our requirements.

### Alternative 3: Clerk

**Pros:**
- Best-in-class developer experience.
- Hosted UI, hosted email, hosted MFA — fastest path from zero to working auth.
- Compliance certifications (SOC 2, HIPAA-ready) come for free.

**Cons:**
- **Users live on Clerk's servers, not ours.** Auth is the highest-trust integration in the product; outsourcing it to a vendor whose pricing, retention policies, and access controls we do not control is a strategic mistake.
- Pricing is per-MAU. At scale, the cost is a recurring tax that makes free-tier-driven growth more expensive than it should be.
- Vendor lock-in on the user object. Migrating off Clerk requires a forensic export plus a custom backfill into our own auth tables.
- Clerk's UI is excellent but opinionated. Customizing to match Lumen's keyboard-first, dense-information aesthetic is fighting the tool.

**Why rejected:** Auth is too important to rent. We will own this layer.

### Alternative 4: Roll our own

**Pros:**
- Total control. Zero dependencies. Maximum learning.

**Cons:**
- WebAuthn alone is several hundred lines of careful code; getting attestation, transports, and counters right is non-trivial.
- TOTP is solved in many libraries; reimplementing it adds risk for no benefit.
- OAuth flows are tedious and edge cases (PKCE, state, nonce, expired refresh tokens) are easy to miss.
- Magic links require careful single-use token semantics.
- Every line of auth code we write is a line we must audit and maintain forever.

**Why rejected:** Auth is the wrong place to express our craft. Better Auth lets us own the data without owning the cryptographic primitives.

## Implementation Notes

- The Better Auth instance reads its database via `betterAuth({ database: drizzleAdapter(db, { provider: 'pg' }) })`. The Drizzle instance is the same one used elsewhere in the app.
- Cookie name is `lumen.session_token`. `Secure`, `HttpOnly`, `SameSite=Lax`. The `Domain` attribute is unset in dev, set to the apex domain in production so the session is shared across the marketing site, app, and any subdomain we publish to in Phase 18.
- Argon2id parameters follow OWASP 2024 recommendations: `memory=19MiB`, `iterations=2`, `parallelism=1`. Cost will be re-tuned annually as hardware improves.
- Passkey registration requires HTTPS in production and uses the localhost workaround in dev.
- TOTP secrets are stored encrypted at rest using `pgcrypto`'s `pgp_sym_encrypt` with a key from `AUTH_SECRET`. Recovery codes are stored hashed with the same Argon2id config as passwords.
- Email verification, password reset, and magic-link delivery all flow through Resend (Phase 7). Until Phase 7 is complete, those flows log to the console in dev and are disabled in production via a feature flag.
- The `hooks.server.ts` session resolver runs first in the handle chain. If the session is invalid, `locals.user` is `null` and protected routes redirect.
- A scheduled job (Vercel Cron, configured in Phase 14) deletes expired sessions and tokens nightly.
