# ADR-003: ORM ‚Äî Drizzle over Prisma, Kysely, raw SQL, and TypeORM

**Status:** Accepted
**Date:** 2026-04-24
**Deciders:** @billyribeiro-ux
**Phase:** 0 ‚Äî Foundation
**Tags:** `database`, `orm`, `data`, `foundation`

---

## Context

With Postgres chosen (ADR-002), the data access layer is the next foundation decision. Lumen's data layer must:

- **Run on the edge.** SvelteKit handlers and webhooks deploy to Vercel's edge / fluid compute. The ORM must work without Node-only APIs and with the `@neondatabase/serverless` HTTP/WebSocket driver.
- **Generate types from schema.** TypeScript strict mode is non-negotiable. Manual type duplication between SQL schema and application types is a source of bugs we will not tolerate.
- **Stay close to SQL.** Lumen's queries include recursive CTEs (graph traversal), `tsvector` full-text search, JSONB operators, and partial indexes. An ORM that hides SQL behind a "rich query builder" wins novice DX and loses everything else.
- **Have a small runtime.** The bundle ships to edge cold-starts; ORM weight directly affects cold-start latency.
- **Be honest about what it does.** Magic ‚Äî implicit lazy loading, automatic eager loading, query rewriting ‚Äî produces queries we cannot predict and cannot tune.

Four candidates evaluated: Drizzle, Prisma, Kysely, raw SQL via `postgres-js`. TypeORM was disqualified up front (decorators-first, anemic Postgres feature support, single-maintainer governance crisis since 2022).

---

## Decision

**We will use Drizzle ORM 0.44.x as Lumen's data access layer.** All schema definitions live in `src/lib/server/db/schema/`. All migrations are generated by `drizzle-kit generate` against those schema files ‚Äî never hand-written, never authored in the database first. Queries are written using Drizzle's query builder for typical CRUD and via Drizzle's `sql` template for anything that requires Postgres-specific features.

Concretely:

- One Drizzle instance per request, constructed from `@neondatabase/serverless`'s `neon()` HTTP client.
- A separate WebSocket-backed instance is constructed lazily for code paths that require multi-statement transactions.
- Schema files are organized by domain (`auth.ts`, `nodes.ts`, `links.ts`, `billing.ts`, ‚Ä¶) and re-exported from `src/lib/server/db/schema/index.ts`.
- Drizzle relations are defined in the same files as the tables they relate. Relations are used for type generation and for the relational query API; they are not used to enable hidden joins.
- Any query that crosses three or more tables, uses a window function, uses a recursive CTE, or uses Postgres-specific syntax (e.g. `ON CONFLICT ‚Ä¶ DO UPDATE`) is written with `sql` templates, not the query builder. Builder is for the 80%; SQL is for the 20%.

---

## Consequences

### Positive

- **Schema is the source of truth.** TypeScript schema files are authoritative; migrations are generated artifacts. There is no migration-first workflow, no SQL-first workflow, no shadow database divergence.
- **Edge-compatible from day one.** Drizzle has zero dependencies that require Node-only APIs. It runs unchanged on Vercel edge, Cloudflare Workers, and Bun ‚Äî relevant if Lumen's deployment topology changes.
- **Bundle weight is negligible.** The Drizzle runtime is small enough that the cost-of-ORM is dominated by the driver, not the ORM itself.
- **Generated types flow through every query.** Selecting a subset of columns, joining tables, returning aggregates ‚Äî all yield correctly inferred result types without manual annotation.
- **The escape hatch is first-class.** Drizzle's `sql` template returns the same typed result shape, so dropping into hand-written SQL never costs us type safety.
- **Migrations are inspectable diffs.** `drizzle-kit generate` writes plain `.sql` files into `drizzle/` that can be code-reviewed like any other file. No black-box migration engine.

### Negative

- **The relational query API is younger than Prisma's `include`.** Some query shapes that are trivial in Prisma (deeply nested includes with conditional filters at each level) require more keystrokes in Drizzle. We pay this cost in exchange for predictable SQL output.
- **No baked-in seeding framework.** Seeding (Phase 2) is something we author ourselves. This is not a Drizzle problem ‚Äî it's a deliberate choice not to take on a vendor's seeding opinion ‚Äî but it is work we own.
- **Migration tooling is less polished than Prisma's.** `drizzle-kit` is good and improving fast; it is not yet at Prisma's level for things like resolving migration drift in shared environments. Mitigated by Neon's per-PR branching: drift cannot accumulate when every PR runs against a fresh branch.
- **Less mainstream than Prisma.** Hiring a candidate with deep Drizzle experience is harder than hiring one with deep Prisma experience. We accept this ‚Äî Drizzle is close enough to SQL that any Postgres-fluent engineer is productive within a day.

### Neutral

- Drizzle is tightly coupled to its driver adapters. Switching from Neon to another Postgres driver requires changing only the adapter import ‚Äî schema and queries are unaffected.
- Drizzle does not generate runtime validators. Validation lives in Valibot (ADR-005); the boundary between persistence and input validation is intentional and explicit.

---

## Alternatives Considered

### Alternative 1: Prisma

**Pros:**
- Most polished DX in the JavaScript ORM space.
- Excellent migration tooling, including drift detection and shadow database workflows.
- Largest community; most tutorials, examples, and AI training data.
- Strong TypeScript type generation.

**Cons:**
- The Rust query engine is heavy. It runs as a separate binary in development and as a WASM module on edge runtimes. Cold-start latency on edge is materially worse than Drizzle's pure-JS approach.
- Generated SQL is opinionated and not always efficient. Common Postgres optimizations (partial indexes used selectively, expression indexes, parametric tsvector queries) require dropping out of Prisma into raw SQL anyway.
- Schema lives in `schema.prisma`, a custom DSL. Schema-as-TypeScript is impossible.
- Recursive CTEs, window functions, and `LATERAL` joins all require `prisma.$queryRaw`, at which point the generated types stop helping.
- Prisma Accelerate (their connection pooler) is a paid managed product layered on top of the ORM ‚Äî fine, but it tilts the ORM toward depending on Prisma's hosted services for production performance.

**Why rejected:** We will hit Prisma's ceiling on day one (recursive CTEs for graph traversal) and pay edge cold-start cost forever for a feature set that Drizzle covers without the runtime weight.

### Alternative 2: Kysely

**Pros:**
- Pure SQL builder; no schema DSL.
- Excellent type inference.
- Tiny runtime.
- Author is rigorous and the project's design discipline shows.

**Cons:**
- No migration tooling. We would author and maintain migrations by hand or layer a separate tool (e.g. `dbmate`, raw `node-pg-migrate`) on top.
- Schema is described per-table in TypeScript interface form; there is no single schema file from which both queries and migrations are derived. The single-source-of-truth property is lost.
- No relational query API. Joins are explicit at every call site, which is fine for small apps and tiring at the scale of Lumen's domain model.
- Smaller community; fewer integrations with downstream tools (validators, dashboards, observability).

**Why rejected:** Kysely is a great query builder but only half of an ORM. We would end up reimplementing the migration and schema-truth pieces, and the resulting bespoke stack is a worse trade than adopting Drizzle whole.

### Alternative 3: Raw SQL via `postgres-js` or `node-postgres`

**Pros:**
- Zero abstraction. Maximum control. Smallest possible runtime.
- Every Postgres feature is available without negotiation.
- No ORM upgrades to manage.

**Cons:**
- Manual type definitions for every query result. With strict TypeScript, this is hundreds of hand-written types that drift from the schema the moment migrations land.
- No migration tooling at all. We become responsible for diffing schema versions.
- Refactors are dangerous. Renaming a column requires grep-driven discipline; the compiler does not catch query/schema mismatches.
- DX cost compounds ‚Äî every new query costs more than it should.

**Why rejected:** Raw SQL is the right choice for a single small service. For Lumen's surface area (dozens of tables, hundreds of queries by v1.0.0), the cost of manual typing is unacceptable.

### Alternative 4: Do nothing (defer the ORM choice; query strings inline)

**Pros:**
- Maximum optionality.

**Cons:**
- Phase 1 builds the schema. Schema authoring without a chosen ORM means writing SQL DDL by hand and then re-authoring it later in whichever ORM we eventually pick. Pure rework.
- The cost of choosing now is low; the cost of not choosing now is the schema is built twice.

**Why rejected:** Foundation phase decisions are not deferrable.

## Implementation Notes

- `src/lib/server/db/index.ts` exports a `db()` factory that returns a Drizzle instance bound to the request's environment. It reads `DATABASE_URL` from `$env/dynamic/private` so the same code path works in dev, preview, staging, and production.
- A `dbWS()` factory returns a WebSocket-backed instance for transactional writes. It is invoked only from code paths that need it (subscription state machine, audit log writes that must atomic-commit alongside the action they log).
- Schema files use `pgTable` exclusively. Common columns (`id`, `created_at`, `updated_at`, `deleted_at`, `organization_id`) are defined as helpers in `src/lib/server/db/schema/_columns.ts` and reused via spread.
- Migrations land in `drizzle/` at the repo root. They are committed and reviewed. CI runs `drizzle-kit migrate` against an ephemeral preview branch on every PR as a smoke test.
- `pnpm db:generate`, `pnpm db:migrate`, `pnpm db:studio`, `pnpm db:push` are the four scripts. `db:push` is dev-only and never used against production.
- The schema is the boundary with Better Auth (ADR-004). Better Auth's required tables (`user`, `session`, `account`, `verification`) are defined in `auth.ts` to match Better Auth's expected shape; Drizzle owns the migrations for them, not Better Auth's CLI.
